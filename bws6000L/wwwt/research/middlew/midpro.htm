Problem:

 

 

Middleware and data integrated problems cut across many disciplines and IT environments. Whenever different types of systems have to be integrated, it results in a need for a middleware or a data integrator. A different type of system may be with respect to hardware, software application, programming language, operating system, database management system, network protocol, data model, desktop application, vendor specific product, and so on. For example, a database application designed in relational model, and another database application designed in object-oriented model may need a middleware to communicate. In some cases, if both applications are designed in a relational model, they may still need a middleware if the relational model vendor’s design is not compatible. Another example related to a manufacturing environment may involve a product design database versus a process design database. A product design database may use a CAD system with certain design attributes relevant to their model, and a process design may use another data model which could be incompatible with the design model resulting in a data integrator problem. This is only a tip of the iceberg when you consider all disperate systems and environments that exist in today’s IT.

 

In addition, this problem gets worst when these systems and environments are constantly evolving and changing by making their predecessors old and obsolete. We still use old applications in today’s IT as they are valuable and requires tremendous efforts and cost to port to new environments.
